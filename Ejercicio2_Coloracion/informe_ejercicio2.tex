\section{Descripción teórica}

El problema de \textbf{\(k\)-coloración} consiste en asignar un color a cada
vértice de un grafo no dirigido \(G=(V,E)\) usando colores del conjunto
\(\{1,\dots,k\}\), cumpliendo que vértices adyacentes tengan distinto color:
\[
\forall (u,v)\in E:\quad \text{color}[u] \neq \text{color}[v].
\]

Relación con número cromático:
\[
G\ \text{es}\ k\text{-coloreable} \iff \chi(G) \le k.
\]

\section{Enfoques de solución}

\subsection{Backtracking}

El enfoque de backtracking construye la coloración de forma incremental,
asignando color al vértice \(v\) y avanzando al vértice \(v+1\). Antes de
avanzar, valida si el color asignado es compatible con los vecinos ya
coloreados mediante la función \texttt{es\_seguro}. Si se detecta un conflicto,
esa rama se descarta inmediatamente (\textit{poda}) y no se sigue explorando.

Este enfoque permite:
\begin{itemize}
  \item encontrar la primera solución válida;
  \item contar todas las soluciones válidas;
  \item reducir significativamente el espacio explorado cuando el grafo es
  denso o cuando \(k\) es pequeño.
\end{itemize}

En el código \texttt{main\_ejercicio2.cpp}, además de contar soluciones,
se guarda la primera coloración válida y se lleva el conteo de nodos explorados
para medir la efectividad de la poda.

\subsection{Fuerza bruta}

El enfoque de fuerza bruta genera todas las asignaciones posibles de colores
para \(n\) vértices, es decir, \(k^n\) combinaciones. Cada combinación se
verifica completa para comprobar si cumple la restricción de coloración.

En esta estrategia no existe poda: aun cuando una combinación ya es inválida en
los primeros vértices, se termina de generar y luego se valida. Por eso su costo
crece rápidamente y se usa principalmente como referencia para comparar
rendimiento y para verificar que el conteo de backtracking sea correcto.

\section{Pseudocódigo completo}

\subsection{Backtracking}

\begin{lstlisting}[style=pseudo]
leer n, k
leer adj[0..n-1][0..n-1]     // matriz de adyacencia
color[0..n-1] = 0            // 0 = sin color
total_soluciones = 0
primera_guardada = falso
nodos_bt = 0

es_seguro(v,c):
  para u en [0..n-1]:
    si adj[v][u] == 1 y color[u] == c:
      retornar falso
  retornar verdadero

bt(v):
  nodos_bt += 1
  si v == n:
    total_soluciones += 1
    si no primera_guardada:
      primera_solucion = copia(color)
      primera_guardada = verdadero
    retornar

  para c en [1..k]:
    si es_seguro(v, c):
      color[v] = c
      bt(v + 1)
      color[v] = 0            // backtrack

bt(0)

si total_soluciones == 0:
  imprimir "No existe una k-coloración válida"
si no:
  imprimir "Total de coloraciones válidas:", total_soluciones
  imprimir "Primera solución:", primera_solucion
imprimir "Nodos explorados:", nodos_bt
\end{lstlisting}

\subsection{Fuerza Bruta}
\begin{lstlisting}[style=pseudo]
total_fb = 0
nodos_fb = 0
total_comb = k^n

verificar_coloracion(col):
  para u en [0..n-1]:
    para v en [u+1..n-1]:
      si adj[u][v] == 1 y col[u] == col[v]:
        retornar falso
  retornar verdadero

para mask en [0..total_comb-1]:
  nodos_fb += 1
  tmp = mask
  para v en [0..n-1]:
    col[v] = (tmp mod k) + 1
    tmp = tmp div k

  si verificar_coloracion(col):
    total_fb += 1

imprimir "Total de combinaciones:", total_comb
imprimir "Coloraciones válidas (fuerza bruta):", total_fb
imprimir "Nodos explorados:", nodos_fb
\end{lstlisting}

\section{Implementación en C++}

El archivo principal es \texttt{main\_ejercicio2.cpp}. Implementa:
lectura de matriz de adyacencia, conteo por backtracking, conteo por fuerza
bruta, primera solución válida y comparación de tiempos.

La medición usa \texttt{REPS = 100000} para reportar tiempo promedio en
microsegundos. Se reporta en \textmu s porque, en estos tamaños de entrada,
si se usa milisegundos (ms) las diferencias entre enfoques pueden verse como
0.00 ms o quedar demasiado comprimidas. En \textmu s la comparación es más
precisa y permite observar claramente el efecto de la poda.

\section{Análisis de complejidad}

\subsection{Complejidad temporal}

\textbf{Fuerza bruta}

Este método genera \(k^n\) combinaciones. Para cada combinación, la función
de verificación revisa pares de vértices adyacentes con la matriz de adyacencia,
lo cual en el peor caso cuesta \(O(n^2)\). Por tanto:
\[
T_{FB}(n,k)=O(k^n\cdot n^2), \qquad S_{FB}(n)=O(n^2).
\]

\textbf{Backtracking}

En el peor caso (por ejemplo, grafos con poca restricción), el árbol de
búsqueda puede acercarse a \(k^n\) asignaciones. En cada intento de asignación,
\texttt{es\_seguro} recorre hasta \(n\) vecinos por usar matriz de adyacencia.
Una cota superior práctica para esta implementación es:
\[
T_{BT}(n,k)=O(k^n\cdot n).
\]

Aunque sigue siendo exponencial, en la práctica suele ser mucho menor por la
poda temprana de ramas inválidas.

\subsection{Complejidad espacial}

Ambos métodos comparten el almacenamiento principal: matriz de adyacencia
\(n\times n\), que domina con \(O(n^2)\). Además, se almacenan arreglos de
colores de tamaño \(O(n)\).

En backtracking se agrega la pila de recursión de profundidad máxima \(n\),
equivalente a \(O(n)\). Entonces, para ambos enfoques, la complejidad espacial
total queda dominada por:
\[
S(n)=O(n^2).
\]

En síntesis: la diferencia clave entre enfoques no está en memoria, sino en
tiempo de ejecución, donde backtracking aprovecha la poda.

\section{Respuestas guía}

1. \textbf{¿Cuántos nodos explora Backtracking vs Fuerza Bruta?}

En el ejemplo \(C_4\) con \(k=3\), backtracking explora 40 nodos y fuerza
bruta 81. La diferencia se debe a que backtracking descarta ramas apenas
aparece una asignación inválida, mientras que fuerza bruta genera cada
combinación completa antes de verificar.

2. \textbf{¿Qué tipo de grafo maximiza o minimiza la poda?}

La poda se maximiza en grafos densos (por ejemplo, \(K_n\)) porque hay muchas
aristas y, por tanto, más probabilidad de conflicto temprano. Se minimiza en
grafos muy dispersos o sin aristas, donde casi cualquier asignación parcial es
válida y el árbol de búsqueda se recorre casi completo.

3. \textbf{¿Qué cambio hacer si solo se quiere saber si existe una coloración?}

Basta con retornar en cuanto se encuentre la primera solución, propagando un
valor de éxito hacia arriba:

\begin{lstlisting}[style=pseudo]
bt_existe(v):
  si v == n:
    retornar verdadero      // primera solución hallada

  para c en [1..k]:
    si es_seguro(v, c):
      color[v] = c
      si bt_existe(v + 1):
        retornar verdadero  // propagar el éxito sin seguir buscando
      color[v] = 0

  retornar falso
\end{lstlisting}

Esto convierte el algoritmo de enumeración en uno de decisión: en el mejor
caso termina mucho antes de explorar todas las ramas.

4. \textbf{Relación con el número cromático.}

El número cromático \(\chi(G)\) es el mínimo número de colores necesario para
colorear un grafo sin conflictos. Por eso, resolver el problema de
\(k\)-coloración equivale a verificar si \(\chi(G)\le k\). En la práctica,
puede estimarse \(\chi(G)\) probando valores crecientes de \(k\) hasta
encontrar el primero que produzca al menos una solución válida.

\section{Conclusiones}

\begin{itemize}
  \item La implementación confirma la corrección de ambos enfoques: en todos
  los casos, backtracking y fuerza bruta reportan el mismo total de
  coloraciones válidas.

  \item Backtracking fue consistentemente más eficiente, tanto en nodos
  explorados como en tiempo de ejecución, debido a la poda de ramas inválidas.

  \item La efectividad de la poda depende de la estructura del grafo. En este
  trabajo varió entre 50.62\% y 98.45\%, mostrando mayor impacto en instancias
  más restrictivas.

  \item El caso \(C_7\) con cuerdas muestra con claridad la diferencia de
  enfoques: cuando no existe solución, backtracking puede detectar conflictos
  pronto, mientras fuerza bruta mantiene el costo de evaluar todo el espacio
  \(k^n\).

  \item Fuerza bruta sigue siendo útil como línea base y como mecanismo de
  verificación cruzada del conteo obtenido por backtracking.

  \item Aunque la poda mejora el rendimiento práctico, ambos algoritmos tienen
  crecimiento exponencial para instancias grandes. Para grafos de mayor tamaño
  conviene estudiar heurísticas de ordenamiento de vértices y métodos
  aproximados.
\end{itemize}
